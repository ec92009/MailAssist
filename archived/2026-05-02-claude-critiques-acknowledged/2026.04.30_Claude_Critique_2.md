## Current Read

The original April 26 defects were mostly fixed. The remaining risk has shifted away from immediate correctness bugs and toward maintainability:

- `src/mailassist/gui/desktop.py` is now a 3085-line single-window class.
- `src/mailassist/bot_runtime.py` and `src/mailassist/background_bot.py` are both over 1000 lines and overlap.
- `review_state.py` is now legacy/test-only weight.
- CLI and GUI action wiring are growing in parallel without a shared action schema.
- Some small safety/support polish remains around settings state, placeholder account text, and fallback heuristics.

## Resolved Since April 26

### GUI

- The former live Gmail test-draft risk is resolved. The visible Gmail preview action is dry-run-only, and real provider-writing actions remain confirmation-gated.

## Current GUI Findings

### 1. `MailAssistDesktopWindow` Is Too Large

`src/mailassist/gui/desktop.py` is 3085 lines with one large `MailAssistDesktopWindow` class. It now owns window chrome, settings, bot controls, recent activity, log dialogs, model checks, provider controls, categories, Elders, and subprocess handling.

This is the dominant GUI risk. The best next step is to extract coherent widgets:

- Settings wizard pages into `gui/settings/`
- Bot controls into `gui/bot_control.py`
- Recent Activity into `gui/recent_activity.py`
- Elders and category editors into small reusable dialog/editor widgets
- Shared style tokens into `gui/style.py`

Severity: High.

### 2. Confirmation Copy Is Repeated

Several actions repeat the same `QMessageBox.question(... Yes|No, default No)` pattern. A small `_confirm_action(title, body)` helper would preserve the default-No safety standard and avoid copy drift.

Severity: Low.

### 3. Settings Dialog Is Non-Modal While Bot Actions Can Still Start

The settings dialog is non-modal. A user can start a bot action while settings are open, before edited values are fully saved. Since bot actions run in a subprocess and reload `.env`, this can create confusing state.

Reasonable fixes:

- Make settings modal, or
- Disable bot-starting controls while `self.settings_open` is true.

Severity: Medium.

### 4. Every Bot Action Starts A Fresh `QProcess`

The subprocess-per-action model is good for crash isolation and Ollama hangs, but it also means every action re-imports the app, re-loads settings, re-instantiates providers, and re-discovers auth/account state.

Do not change this reflexively. Consider a long-lived worker only if startup tax or progress lag is visible during Magali deployment.

Severity: Medium.

### 5. Bot/Ollama UX Uses Several Timers

`bot_heartbeat_timer`, `bot_timeout_timer`, `ollama_test_countdown_timer`, and delayed `singleShot` refreshes coordinate related state. A `BotActionTimers` helper would make leakage or missed-stop bugs easier to reason about.

Severity: Low to Medium.

### 6. Banner, Pill, And Progress Bar Are Separate State Surfaces

The GUI communicates state through a banner, status pills, and the progress bar. A single `set_app_state(level, message, progress=None)` style entry point would reduce inconsistent updates.

Severity: Low.

### 7. Desktop Tests Exercise The Full Window

`tests/test_desktop_layout.py` is large because there are no smaller panel widgets to test directly. Extracting panels would let tests become narrower and less brittle.

Severity: Low.

## Current Architecture Findings

### 1. `review_state.py` Is Legacy/Test-Only Weight

`review_state.py` is still 1048 lines, but it is no longer imported by production source paths. It remains mainly for tests and legacy compatibility.

Options:

- Delete it and its tests if the old review path is fully retired.
- Or reduce it to the small compatibility surface that is genuinely needed.

Severity: High.

### 2. Drafting Prompt Rules Are Still Duplicated

Prompt/rules code was reduced from four duplicated sites, but drift remains across `drafting.py`, `background_bot.py`, and legacy review support.

If `review_state.py` goes away, remove the duplicate review-style prompt builders with it. Otherwise, centralize shared rules in one helper.

Severity: Medium.

### 3. `bot_runtime.py` And `background_bot.py` Overlap

`bot_runtime.py` is the CLI/action dispatcher, while `background_bot.py` owns live watcher and reply construction, but both touch live state and draft assembly. A future split could make:

- `bot_runtime.py`: argparse and dispatch only
- `live_pipeline.py`: watcher pass and state writes
- `replies.py`: recipients, reply metadata, quoted context, safe recipient helpers
- `organize.py`: Gmail/Outlook category organization

Severity: Medium.

### 4. Provider Modules Duplicate Shared Shape

`providers/gmail.py` and `providers/outlook.py` are both large and implement similar cross-cutting behavior. `providers/base.py` exposes the contract but not much shared helper behavior.

Potential shared helpers:

- account-email memoization
- recipient de-duplication
- reply metadata shape
- account/from-address selection conventions

Severity: Medium.

### 5. `load_settings()` Falls Back To `Path.cwd()` Without Warning

In non-frozen dev mode, settings root falls back to the current working directory. That is useful, but silent. Add a warning when `.env` is missing at the resolved root, or eventually require `MAILASSIST_ROOT_DIR` for non-frozen invocations.

Severity: Low.

### 6. Stale Ignored `__pycache__` Directories May Remain Locally

Deleted `core/` and `storage/` sources can leave ignored bytecode caches on disk. They are harmless, but can confuse grep/IDE audits. A dry-run `git clean -fdX src/` can show what would be removed before cleaning.

Severity: Low.

## Current Miscellaneous Findings

### 1. `append_signature_to_body` Is Still Duplicated

`drafting.py` and `review_state.py` both define `append_signature_to_body`. Production uses the `drafting.py` implementation. If `review_state.py` remains, it should import the shared helper rather than redefining it.

Severity: Medium.

### 2. Fallback Classification Uses Broad Substring Matching

The heuristic fallback still scans broad text with substring tests. False positives are possible for words like `digest`, `urgent`, `asap`, and `unsubscribe`.

Better shape:

- Give sender and subject higher confidence.
- Use word boundaries for body matching.
- Avoid quoted-history body matches where possible.

Severity: Low to Medium.

### 3. Batch Candidate Parsing Fails The Whole Batch When One Block Is Missing

`parse_batch_candidate_response` raises if any expected block is absent, causing all threads in the batch to fall back to per-thread generation. It would be more efficient to keep successfully parsed blocks and retry only missing ones.

Severity: Low.

### 4. Review State Still Lives Under `data/legacy/`

This is fine if the review module is being deleted. If the review path remains active in any form, the path name is misleading and should move back to `data/review-inbox.json`.

Severity: Low.

### 5. `you@example.com` Can Still Leak In Synthetic Paths

Mock fixtures legitimately use `you@example.com`, but synthetic controlled draft/report paths should pass the resolved account email where available. This is low-risk but easy polish.

Severity: Low to Medium.

### 6. Plain And HTML Signatures Can Drift

The wizard normally keeps plain and HTML signatures aligned, but direct `.env` edits can make the two MIME parts disagree. A future fallback could derive HTML from plain text when the HTML signature is absent or stale.

Severity: Low.

### 7. CLI And GUI Bot Action Flags Are Hand-Spelled Twice

`bot_runtime.py` defines argparse flags, while `gui/desktop.py` assembles corresponding CLI arguments by hand. A shared `BotActionSpec` dataclass would reduce typo drift as actions grow.

Severity: Low.

## Priority Order

1. Split `MailAssistDesktopWindow` into smaller panel widgets.
2. Decide the fate of `review_state.py`; delete or shrink it.
3. Disable bot actions while settings are open, or make settings modal.
4. Move duplicated signature/prompt/reply helpers into shared modules.
5. Add a warning for dev-mode `load_settings()` fallback to cwd.
6. Clean up `you@example.com` synthetic-path leakage.
7. Consider `BotActionSpec` only if CLI/GUI action drift starts causing bugs.

## Do Not Change Lightly

- Keep provider writes explicit and confirmation-gated.
- Keep JSONL bot events as the stable subprocess protocol.
- Keep local Ollama as the trust posture.
- Keep subprocess isolation unless Magali deployment proves startup overhead is a real problem.
- Keep PySide6 unless Windows packaging blocks an actual ship date.
